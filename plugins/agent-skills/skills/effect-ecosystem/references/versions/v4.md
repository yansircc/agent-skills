# Effect v4 Notes

Use local `node_modules/effect` as API truth. Check `references/v4-migration.md`
before applying v3 imports or ecosystem package names. Treat package
reorganization as version-specific reference material, not as a rule fact.

Current pinned acceptance target: `effect@4.0.0-beta.84`. v4 capability is
accepted only when typed API, LSP diagnostics, browser bundle, Worker/D1
dry-run, and OTel peer closure all pass. Package existence alone is not
evidence.

Known v4 beta deltas:

- Low-level DI uses `Context.Service`, not v3 `Context.Tag` examples.
- `HttpApiEndpoint.get(name, path, { success })` replaces chained
  `.addSuccess(...)` for v4 examples.
- React state integration remains a gap until a v4-compatible React adapter is
  proven; browser runtime support is not the same fact.

## v4 OTel Peer Closure

For pinned `effect@4.0.0-beta.84`, `@effect/opentelemetry` requires the peer
closure declared in `contracts/effect-capabilities.json`.

This rule is `beta-pinned`: if the v4 beta or peer closure is refreshed, run
`make verify-v4-acceptance` before keeping missing peers as `EFF323`. Until that
proof is refreshed, the scanner must emit `effect-capability-proof-gap` instead
of a deterministic finding.
